对于与服务器一起运行的servlet

在web.xml中，写清楚url和类全名，启动时解析xml, 放到一个类似Map的容器中，之后进行匹配。匹配到时，通过反射即可得到注册的servlet。

这实际上是类似嵌入式Tomcat, 但更多的情况下，是先写Servlet, 然后引入内嵌Tomcat运行。

对于加载外部war包

操作方式应对和前面一样，通过扫描解压得到的web.xml，根据路径和Servlet全名进行注册。

但问题在于，是否可以直接反射获取外部的Servlet

肯定是不行的。因为这些类还没有被加载。

这里引出一个重要的问题：Tomcat类加载机制

类加载

类加载是指将类的字节码从文件系统或网络中加载到内存中，并转换为 java.lang.Class 对象的过程。在 Java 中，类加载是由类加载器（ClassLoader）动态完成的。

Java 中的类加载是 动态 进行的，即在程序运行时按需加载类。

java从哪里加载类，本地写的Servlet为什么可以被加载

Java 使用类加载器从上述不同来源加载类。在应用程序启动时，JVM 会根据不同的路径来搜索和加载类文件。

  • 启动类加载器（Bootstrap ClassLoader）：

它从 JDK 的 rt.jar 等核心类库中加载系统级类，如 java.lang.、java.util. 等类。
加载这些类时，不需要从 classpath 路径查找，而是直接从 JDK 自带的类库中读取。

  • 扩展类加载器（Extension ClassLoader）：

它从 JVM 的扩展目录（通常是 /lib/ext）中加载扩展的 JAR 包。
如果你在这类目录下放置了类库，扩展类加载器会优先加载。

  • 应用类加载器（AppClassLoader）：

它从 classpath 指定的目录中加载应用程序的类，包括你编写的代码以及你引用的第三方类库。
通过运行 Java 程序时指定的 -cp 或 -classpath 参数，JVM 知道应该从哪些路径搜索类。

而

这个机制实际上避免了核心类库被重写。

思考：手写java.lang.String 类，编译，会发生什么

而Tomcat启动时，本身的类，以及写在一起的Servlet类由AppClassLoader加载。那么，引入的war包没法加载。要想引入的war包里的类被正确加载，可以自定义一个类加载器，重写相关方法。实际上，就是从war包所在路径，加载.class文件即可。
